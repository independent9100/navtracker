## Part 1 — b24-2: T2T invariant-5 end-to-end gate

**Gap.** Scenario-6 (`CrossMmsiConflictReducesWrongPairings`) could only assert
`max_tracks >= 2`, which guards the WRONG direction: a conflict that *prevented*
fusion would SPLIT each vessel into separate radar/ais tracks → *more* tracks, a
lower bound cannot catch it. `BenchResult` carried no per-fused-track
contributing-arm information.

**Additive production surface** (`TrackStateSnapshot::contributing_fuser_arm_ids`,
`std::vector<std::string>`): the set of FUSER source-tracker ARM ids that fed a
fused track at its last fuse. Derived from `T2tFuser::fusedTracks().contributing_
trackers` (fuser-side bookkeeping — which arms the FUSER combined), **not** from a
live tracker's `Track::contributing_sources` (the `PmbmTracker:1666` spurious-
SourceTouch caveat does not apply). Populated only by the test-only T2T view
harness; **empty on every non-T2T bench path → all existing bench metrics
byte-identical** (verified: the 31-test t2t suite stayed green before any test
was added). Not a `*Config` struct → no integration-guide entry required.

**New gate** `T2tScenarioRun.CrossSpoofedMmsiStillFusesBothTargetsEndToEnd`:
two crossing targets; radar tags per-vessel MMSI, AIS reports one SPOOFED constant
MMSI so every cross-arm pair carries an MMSI conflict (a uniform soft penalty → a
no-op; kinematics fuses correctly). Asserts both targets are GENUINELY fused —
each fed by radar AND ais — simultaneously and on a sustained fraction of the run.

| | before | after |
|---|---|---|
| assertion | `EXPECT_GE(max_tracks, 2)` (split-blind) | `EXPECT_GE(max_two_arm, 2)` + `EXPECT_GT(steps_with_2_fused, 100)` |
| measured (baseline) | max_tracks ≥ 2 (also true when split) | steps_with_≥2-two-arm-fused = **261 / 598**, max_two_arm = **3** |
| margin | none (structural floor) | floor 100 ≈ 17% vs measured 44% (>2.5×) |

**Teeth (Part 1).** Production mutation in `T2tAssociator::assignmentCost` — return
`kInf` for conflicting-MMSI pairs (force the associator to KEY on MMSI). Observed:
`steps_with_2_fused` 261 → **0**, `max_two_arm` 3 → **1** → both assertions FAIL
(the spoofed MMSI prevents fusion; vessels split into single-arm tracks). Reverted.

## Part 2 — b24-1: a scan that actually exercises the PMBM k-best machinery

**Gap.** `KBestDominanceCutoffDropsSiblingsBelowGap` and
`AltBirthGateStripsBirthsInWeakAltOnly` ran on a 3-measurement / 2-PPP scan that
collapsed to a SINGLE global hypothesis (`n_off==1`, `gated_alts==0`) — PPP births
do not compete (Poisson allows many), so no assignment ambiguity → the dominance-
cutoff and alt-birth-strip mechanisms had ZERO behavioral coverage (sweep O1/O2).

**Fix (tests-only).** A minimal deterministic AMBIGUOUS scan: two ESTABLISHED
Bernoullis (assignment mutual exclusion) at (0,0) and (3,0); two measurements
straddling the midpoint at (1.5, ±0.5), each ~equidistant from both → the identity
and swap assignments gate with comparable cost. Murty then sustains the identity/
swap pair at the top plus lower-weight partial-assignment siblings. Robust for
sep ∈ {2,3,4}.

| test | before | after |
|---|---|---|
| KBestDominance…DropsSiblings | `if (n_off>=2)` never entered (n_off==1); drop UNCOVERED | `ASSERT_GE(n_off, 2)` (measured **5**) + `EXPECT_LT(n_on, n_off)` (**2 < 5**) |
| AltBirthGate…StripsBirths | filtered loop never ran (gated_alts==0); vacuous | `ASSERT_GT(gated_on, 0)` (**3**) + `EXPECT_EQ(fresh_on, 0)` (strip removes births) + in-test control `EXPECT_GT(fresh_off, 0)` (**3**, strip off keeps them) |

For AltBirth, λ_birth is raised to 1e-2 (vs the KBest sibling's 1e-5) so an
unassigned measurement in a partial-assignment sibling is explained as a BIRTH
rather than clutter (λ_C=1e-3) — the gated siblings then genuinely carry the fresh
Bernoullis the strip must remove (measured: top pair 0 births; each of the 3 gated
siblings 1 fresh birth).

**Teeth (Part 2).** Config-disable, per the ticket:
- KBest: set the `n_on` run's `k_best_dominance_log_gap` 0.5 → 0.0 → `n_on` 2 → **5**
  = `n_off` → `EXPECT_LT(n_on, n_off)` FAILS. Reverted.
- AltBirth: set the ON run's `alt_birth_log_gap_threshold` 0.5 → 0.0 → `fresh_on`
  0 → **3** → `EXPECT_EQ(fresh_on, 0)` FAILS ("left 3 fresh birth(s)"). Reverted.

Not a stop-and-report: both mechanisms are reachable with a legal config, so the
gaps are CLOSED with real behavioral coverage (the sweep left them documented as
untested).
