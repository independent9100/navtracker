# Perf round 2 — fresh profile + per-scan latency + safe-class fixes (post-Murty)

Ticket: `docs/superpowers/plans/2026-07-06-perf-round2-ticket.md`.
Branch: `perf-round-2` (worktree `../navtracker-perf2`, off master `ceee0bf`).
Date: 2026-07-06. Origin: the Murty K=1 early exit (`45a504d`) removed the old
85 % bucket and made the 2026-07-05 profile obsolete.

**One-line result:** post-Murty there is **no single dominant safe-class lever**.
The tracker cost is now spread across the IMM per-mode measurement update
(gate / likelihood / `predictMeasurement` / Eigen `S⁻¹`,det). The only clean
safe wins are harness-side (`--fast-metrics`, −10.3 % on raw; per-scan latency
instrumentation). Code-only safe fixes **cannot** bring the raw workload under
the 57 s (≥5×) gate — but the new per-scan measurement shows the tracker keeps
up **scan-by-scan** even at raw density (max scan 101.7 ms < 148 ms interval).

## Method / honesty notes

- **Wall numbers**: clean Release build (`build/`), single-threaded, warm
  (one warm-up run discarded per batch; CPU-frequency drift documented). This
  session ran ~6 % (dec) / ~15 % (raw) warmer than the logged 2026-07-06
  baselines — **compare within-batch, not against the logged absolute walls.**
- **perf was unavailable** (`kernel.perf_event_paranoid = 4`, and `sudo` in this
  session has no TTY to accept a password). Profiling used **gprof** on a
  separate `-pg` build (`build-pg/`, `-O2 -pg -fno-omit-frame-pointer`); all wall
  numbers are from the clean non-instrumented build.
- **gprof is bucket-level ranking only, NOT percentage-precise.** At `-O2` its
  mcount instrumentation over-attributes call-heavy Eigen code and its symbol
  folding / inlining mis-attributes callers. Two demonstrated distortions this
  round:
  - It attributed the harness GOSPA/OSPA/T-GOSPA Hungarian at **27 % (dec) /
    46 % (raw)**; the honest with/without measurement is **1.7 % / 10.3 %** — a
    ~4–15× over-attribution.
  - It reports `ImmEstimator::gate → ImmEstimator::initiate` (34.7 M / 118 M
    calls) and a `CoordinatedTurn::~CoordinatedTurn()` bucket of 140 M / 417 M
    calls. The source of `gate()` calls neither `initiate()` nor constructs a
    `CoordinatedTurn` — these are inlining + identical-code-folding artifacts.
    Read them as "the IMM per-mode Eigen temporaries are hot", not literally.
- Where a number is **accurate** (chrono / with-vs-without wall) vs **gprof
  relative ranking** is called out below.

### Tracker-internal ranking (gprof relative — %s inflated, ordering trustworthy)

1. **IMM per-mode measurement update** — the dominant tracker bucket.
   `ImmEstimator::gate` (17.5 M dec / 59.5 M raw calls) + `predictMeasurement`
   (41 M / 130 M) building `H`, `S = H·P·Hᵀ + R`, then `S⁻¹` and `det(S)` via
   Eigen LU/LDLT, + `measurementResidual`. The folded "`CoordinatedTurn::~`"
   dtor bucket (140 M / 417 M) is the Eigen temporaries in this path. Scales
   with **measurements × Bernoullis × modes**.
2. **PMBM merge/prune** — `bhattacharyyaState` (7.6 M / 22 M), ~2.6 % / 1.6 %.
3. **PPP birth** — `buildAdaptiveBirthCandidates` (1998 calls), ~1.6 % / 1 %.
4. **Tracker assignment** — `murtyKBest → hungarianAssignment`: **1998 calls,
   ~0.7 s (dec) / ~4.6 s gprof-inflated (raw)**. The old 85 % bucket is **gone**.
   Of the 22 479 total `hungarianAssignment` calls, only 1998 are the tracker;
   the other 20 481 are harness OSPA (3267) / GOSPA (3560) / T-GOSPA (7120) /
   gospa-decomp — i.e. the "Hungarian" line is now mostly the metric harness.

### Named top lever for the RAW workload

The **IMM per-(measurement × Bernoulli × mode) measurement-update linear
algebra** in `gate()` / `logLikelihood()` / `predictMeasurement()`. Raw is 3.4×
the dec wall for 1.83× the rows because per-scan cost ~ measurements ×
Bernoullis and the phantom Bernoulli population is ~2× denser at raw density
(card_err 100.5 vs 48.8). Everything else is secondary.

## Step 3 — safe wins implemented

### Fix 1 — per-scan latency instrumentation (harness)
Commit: see below. Byte-identical (adds columns; determinism check extended to
exclude the wall-derived `scan_proc_ms_*`, exactly as `wall_seconds` already was
— `scan_interval_s`/`n_scans` are data-derived and stay checked). No wall

### Fix 2 — `--fast-metrics` bench flag (harness; probe follow-up (b))
Skips per-cell accuracy/consistency scoring (OSPA/GOSPA/T-GOSPA/RMSE/NEES/NIS),
emits only wall + per-scan latency. Tracker still runs in full.
Default-off ⇒ **byte-identical** (verified: default run reproduces gospa 104.262
etc.; only `wall_seconds` differs).

## Priced findings — NOT implemented (arbiter decisions)

1. **Position2D `H`-elision fast-path** in `predictMeasurement`/`gate`: for a
   position measurement `H = [I₂ | 0]`, so `H·P·Hᵀ = P.topLeftCorner(2,2)`
   bit-for-bit (identity products; `a + 0.0·x = a` in IEEE), and `predictMeasurement`
   would skip allocating a mostly-zero dynamic `H` per call (41 M / 130 M calls).
   Projected ~5–8 % tracker. **Plausibly bit-identical but blast radius = every
   estimator and every measurement model** — it must be proven byte-identical
   per-model before it counts as safe-class. Recommended as its own small
   safe-fix ticket (implement → full suite → byte-compare both workloads).
2. **Coarse position-box gate prefilter** before the per-mode Mahalanobis gate:
   `gate()` is the top tracker bucket (17.5 M / 59.5 M full evaluations). A cheap
   bounding-box reject would cut most of them. **Larger projected win, but it
   changes which (track, measurement) pairs gate → result-affecting.** Arbiter
   decision; needs accuracy pricing, not a safe fix.
3. **Sparse/gated LSAP** (probe follow-up (a)): **deprioritised** — the tracker's
   own assignment is now ~1998 calls / a few percent. The dense-matrix waste it
   targeted lived in the old 85 % bucket, which the Murty fix already removed.

## Verdict (acceptance Q4)

**Can code-only SAFE fixes bring the RAW workload under 57 s (≥5× margin)? — NO.**
`--fast-metrics` brings raw to 146.5 s, but that is bench wall; the deployment-
relevant floor is the tracker `processBatch` sum ≈ **147 s**. Removing *all*
harness overhead still leaves ~2.6× over the 57 s budget, and safe-class
(bit-identical) fixes cannot materially cut the FP-sensitive IMM Eigen work.
Reaching 57 s needs a ~2.6× tracker speedup, achievable only via a
**result-changing** algorithmic change (front-end extraction/decimation, or
aggressive gating/pruning). **Front-end extraction stays deployment-MANDATORY
for the ≥5× batch-throughput margin gate** — unchanged from 2026-07-06.

**Refinement (new this round, for the arbiter):** the ≥5× gate is a
*batch-throughput safety margin*, not a *streaming-feasibility* bound. Per-scan,
the **worst** scan (101.7 ms) fits inside one scan interval (148 ms) with 1.45×
margin at raw density — the tracker keeps up scan-by-scan without extraction.
So extraction is mandatory for the throughput margin, **not** because live
tracking would stall. The north-star verdict cell is left unchanged; this nuance
is flagged for the arbiter.
