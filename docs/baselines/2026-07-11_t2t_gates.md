# T2T two-tracker-view gates — 2026-07-11

Milestone-3 scenario results for `navtracker_t2t`. Branch `t2t-fusion`. These
numbers are from the engine **after** the M2 adversarial-review fixes
(scan-batched cycle, reporter-driven M-of-N, birth pruning + co-located-birth
clustering, coast-clears-current). The Checkpoint-2 load-bearing number is the
CI-vs-naive NEES ratio in scenario 2.

## How to reproduce

```
# build (worktree ../navtracker-t2t, foxglove off)
cmake --build build -j --target navtracker_tests
# run the sim-gated T2T scenarios against the main-tree fixtures
SIMMS_DIR=/home/andreas/workspace/navtracker/tests/fixtures/sim_multisensor \
  ./build/navtracker_tests --gtest_filter='T2tScenarioRun.*'
```

Base scenario: `sim_ms_headon` (2-vessel head-on, radar+AIS), seed 0. Per-arm
tracker: the canonical `imm_cv_ct_mht` config, one MHT instance per view. Views
are the full scenario's measurements filtered by `SensorKind` in-process (no
`SIMMS_RADAR_ONLY` env global). Scoring: `computeMetrics` (GOSPA c=20) and
`computeNees` (assoc gate 100 m) on the fused, confirmed-only track set vs the
scenario's independent truth. Deterministic (fixtures frozen; fusion is a pure
function of the two per-view `BenchResult`s).

## Checkpoint-2 number — scenario 2, CI vs naive NEES

The double-counting hazard: two trackers that both fused the **same AIS stream**
have correlated errors. Naive (independence-assuming) fusion counts that shared
AIS twice → the fused covariance is too small (overconfident). CI does not.

Position NEES (2-DoF; consistent ≈ 2), `sim_ms_headon` s0:

| Arm split | Rule | NEES n | NEES mean | NEES median | cov95 | naive/CI mean |
|---|---|---|---|---|---|---|
| **Maximal** (A=AIS, B=AIS — both fuse the same AIS) | CI | 894 | 0.482 | 0.206 | 1.00 | — |
| | naive | 894 | 0.824 | 0.255 | 0.984 | **1.71×** |
| Partial (A=radar+AIS, B=AIS — radar dilutes A) | CI | — | — | 1.283 | — | — |
| | naive | — | — | 1.221 | — | ~1.0 (median) |

**Reading it honestly:**

- **Maximal sharing** is the clean gate: two trackers on the same AIS. Naive
  fusion is **1.71× more overconfident** than CI (mean NEES) — close to the
  theoretical 2× for fusing a duplicated estimate. The below-2 value is because
  birth/pairing transients leave some scans with a single contributor (ratio 1
  there), diluting the aggregate; on the scans where both arms genuinely
  co-fuse the ratio is ~2. CI stays consistent throughout.
- **Absolute NEES < 1** (CI median 0.21, cov95 1.00) means the per-tracker AIS
  covariances are *conservative* (under-confident) on this sim. So the
  double-counting shows as a **confidence-inflation ratio**, not a χ²
  band-violation: halving an already-loose covariance still lands inside the
  band. If the inputs were calibrated (NEES ≈ 2), the same 1.71× would push
  naive to NEES ≈ 3.4 — a clear band violation. The ratio is the invariant
  evidence; the band-violation is a consequence only under calibrated inputs.
  This is why the gate asserts the ratio (naive mean ≥ 1.4× CI), not a band.
- **Partial sharing** (the ticket's literal A=radar+AIS / B=AIS): radar
  dominates A, so the shared-AIS correlation is small and naive ≈ CI. This is a
  characterization row, not a gate — it shows the operational magnitude on a
  radar-dominated tracker, where double-counting is mild.

**Ways to improve / follow-ups** (feed the algorithm-doc §4):
- Input-covariance calibration on the sim (or a scenario with calibrated
  trackers) would let the gate assert a χ² band violation directly.
- The partial→maximal spread quantifies "double-counting severity scales with
  shared-sensor dominance" — worth a sweep across arm mixes in M4.
